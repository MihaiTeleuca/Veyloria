Documentation

Set up Veyloria with confidence.

This documentation page is written for server owners, administrators and support staff who want to launch Veyloria in a clean and controlled way. It explains the recommended setup flow, permissions, channel structure, economy configuration, module launch order and support expectations before members begin using the bot heavily.

The goal is to keep the product experience clear. Veyloria should not feel like a bot that was dropped into a server without context; it should feel like a system that has been configured, explained and connected to the support hub properly.

Quick start

From invite to first working setup.

Start by inviting Veyloria with the official invite link, then place the bot role high enough in the role hierarchy for the actions it needs to perform. The bot does not need to sit above staff roles unless it is expected to manage roles below them, but it must be above any role it needs to assign, remove or check in controlled systems.

After the bot joins, create or verify the channels used for economy commands, logs, support, reports and documentation. Do not open every module immediately. A cleaner launch starts with setup, profile, wallet, bank and basic income before advanced systems such as market, business, clans, racing or pets are made public.

1

Invite the bot

Use the official invite button and make sure the server owner or a trusted administrator completes the authorization.

2

Place the role

Move the Veyloria role above the member roles it needs to manage, while keeping owner and staff roles protected.

3

Run setup

Use the setup and configuration commands before allowing members to use the economy publicly.

Recommended first commands: v!setup v!permissions v!config v!setchannel #bot-commands v!setlog #economy-logs

Permissions

Give enough access to work, not enough to become unsafe.

Veyloria should have the permissions needed to send readable messages, use application commands, read command context, create support responses and display embeds. If your server uses role panels, ticket panels or controlled access roles, the bot role must also be above the roles it is expected to manage.

Avoid giving Administrator by default. Administrator is convenient, but it makes troubleshooting harder and gives more power than most economy features require. A professional setup uses specific permissions and keeps dangerous capabilities limited to trusted staff workflows.

View ChannelsRequiredNeeded so Veyloria can read and respond inside configured bot, support and log channels.
Send MessagesRequiredNeeded for command responses, support instructions and public panels.
Embed LinksRequiredRecommended for clean economy cards, profiles, logs and documentation style responses.
Read Message HistoryRecommendedUseful for context, moderation, support and interaction-heavy channels.

AdministratorNot recommended by defaultOnly use temporarily for testing if needed, then replace it with explicit permissions.
Role hierarchy matters

If a role command does not work, the most common cause is not the command itself. Check whether the Veyloria role is above the role it is trying to manage.

Channel setup

Prepare the server before members start using the economy.

A clean Veyloria server should separate public information, command usage, logs, reports and staff review. This prevents members from running economy commands inside rules or announcements, keeps logs readable and makes support cases easier to investigate.

A

Public command channel

Use a dedicated channel such as #bot-commands for normal member commands. This keeps the rest of the server clean.

B

Economy logs

Create #economy-logs for important financial actions, staff corrections, trades, suspicious activity and module changes.

C

Support area

Use tickets or support channels for setup issues, premium questions, command problems and private server-specific cases.

D

Documentation area

Keep guides, FAQ, official links and known issues visible so members do not open tickets for questions already answered.

Configuration

The first configuration should be simple and stable.

Do not configure every advanced module at once. Start with the base economy and verify that users can create profiles, view balances, deposit money, withdraw money and claim basic rewards. Once the base works, add market, business, resources and progression modules gradually.

Suggested baseline: Prefix: v! Currency name: Veyl Currency symbol: VL$ Command channel: #bot-commands Economy logs: #economy-logs Support path: ticket system or support hub Public modules: profile, balance, bank, daily, work Locked modules at launch: market, business, clans, racing, pets

After the baseline is stable, server staff can decide whether to open more advanced systems immediately or treat them as staged releases. A staged release is usually cleaner because it gives members time to understand the economy before complex trading, business and competition systems become active.

Launch order

Recommended module release path.

A good economy launch avoids overwhelming users. Start with identity and simple money movement, then add active income, then market systems, then long-term progression. This helps staff detect issues early and prevents the economy from becoming chaotic on the first day.

StageModulesPurpose
Stage 1Profile, wallet, bank, daily, workGives users the basic economy loop and lets staff confirm that balances are stable.
Stage 2Inventory, shop, pay, net worth, statementsAdds item ownership, transfers and basic tracking without introducing complex markets yet.
Stage 3Market, auctions, trading, price checksOpens user-to-user economy movement and requires stronger logs and support readiness.
Stage 4Business, properties, jobs, careersAdds long-term income and advanced progression once the economy base is stable.
Stage 5Clans, racing, crafting, farms, petsExpands the ecosystem into competition, lifestyle systems and collection paths.

Support flow

How users should ask for help.

Support is faster when the user provides context from the beginning. A ticket that only says “the bot does not work” is not enough. A useful report explains the command, the expected result, the actual result, the channel where it happened and whether the issue happens once or every time.

Recommended support format: Command: What I expected: What happened: Channel: Server setup change before issue: Screenshot or error text: Does it happen every time:

Public questions should stay in community help if they are simple and not private. Tickets should be used for server-specific configuration, premium access, suspicious activity, data corrections or issues that require staff review.

Safety reminder

Support should never require Discord tokens, bot tokens, passwords, recovery codes, payment credentials, hosting credentials or private keys. If someone asks for those, do not continue the conversation and report it through the official support server.

Troubleshooting

Common problems and what to check first.

Check whether Veyloria can view the channel, send messages, embed links and use application commands. Also confirm that commands are being used in the configured command channel if the server restricts economy usage.
Check the role hierarchy first. The Veyloria role must be above the roles it is trying to assign or remove. Also verify that the bot has Manage Roles permission only if role management is required.
Make sure the bot was invited with the applications.commands scope and that the server has allowed the command integration. Sometimes application commands also need time to sync.
Check whether the economy module is enabled, whether the command channel restriction is active and whether the user has started their profile with the required start command.
Review the statement or economy logs if available. Staff should check recent transfers, purchases, admin corrections, market actions and any module changes before assuming data is broken.
Check whether the user has the correct premium role, whether premium access is connected to the correct server and whether the premium information page has the final activation process.

Final checklist

Before announcing Veyloria publicly.

Use this checklist before making the bot available to all members. A careful setup prevents most support problems and makes the first impression much better.

The official invite link has been tested by the server owner or a trusted administrator.
The Veyloria role is high enough to manage only the roles it actually needs.
The bot can view, send messages, embed links and use application commands in the correct channels.
Economy commands are restricted to the correct bot command channel if the server uses restrictions.
Economy logs, moderation logs and support ticket areas are prepared before public launch.
The baseline economy modules have been tested: profile, balance, bank, deposit, withdraw, daily and work.
Advanced modules such as market, business, clans, racing and pets are either tested or intentionally locked for later release.
Rules, support policy, community guidelines, privacy policy and terms pages are linked from the website and support server.
Staff knows how to handle reports, suspicious activity, premium questions and configuration issues.
